Why is this an issue?

When testing non-Boolean values (like integers, pointers, or floating-point numbers) in conditions, it's ambiguous and can be confusing whether you're testing for truthiness (non-zero) or falsiness (zero).

Making such tests explicit improves code readability and clarifies intent:

  • Clearer what value is being tested
  • Prevents accidental bugs from implicit conversions
  • Makes the code more maintainable and easier to review
  • Reduces cognitive load for readers

Example:

    int count = 5;
    int *ptr = malloc(sizeof(int));
    
    // Noncompliant - implicit comparison with zero
    if (count) {
        printf("Count is non-zero\n");
    }
    
    if (!ptr) {
        printf("Pointer is NULL\n");
    }
    
    if (count < 0) {
        printf("Count is negative\n");
    }
    
    // Compliant - explicit comparison with zero
    if (count != 0) {
        printf("Count is non-zero\n");
    }
    
    if (ptr == NULL) {
        printf("Pointer is NULL\n");
    }
    
    if (count < 0) {
        printf("Count is negative\n");
    }
    
    // Explicit comparisons are already acceptable
    if (count == 0) {      // OK - already explicit
        printf("Count is zero\n");
    }

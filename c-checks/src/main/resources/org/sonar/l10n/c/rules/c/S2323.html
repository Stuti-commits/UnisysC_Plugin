Why is this an issue?

When a line of source code ends with a backslash (\), the compiler merges it with the next line during translation phase 2 — a process known as line-splicing. When this occurs inside a // single-line comment, the following line of code is silently treated as part of the comment and never executed, leading to unintended code removal and potentially undefined behavior.

Noncompliant code example

bool isSpecialCharacter(char c)
{
    // Characters considered special: [ ] \
    return 91 <= c && c <= 93;   // Noncompliant - return statement is part of the comment
}

Compliant solution

bool isSpecialCharacter(char c)
{
    // Characters considered special: [ ] (backslash)
    return 91 <= c && c <= 93;   // OK - no line-splicing in comment
}

Limitations

Note that compilers may also strip trailing whitespace between a backslash and the newline character, meaning trailing spaces do not reliably disable line-splicing. This behavior is standard-compliant since C++23. The companion rule S1916 detects trailing whitespace after \ and should be enabled alongside this rule.
